Постановка задачи

От сырых объявлений до воспроизводимого аналитического сервиса

Цель проекта — построить не статичный notebook, а полноценный data-service: данные собираются и сохраняются в базе, backend отдаёт API, frontend показывает аналитику, а ML-модель рассчитывает стоимость квартиры по параметрам пользователя.

Основная демонстрация выполняется через Streamlit dashboard: данные, карта, прогноз цены, мониторинг и качество модели.

Архитектура

Система разделена на слои

Архитектура соответствует идее DataPulse: ingestion, storage, ML pipeline, backend, frontend и deployment существуют как отдельные, проверяемые части.

Data
Domclick/CIAN-потоки, нормализация, отчёты качества.
Storage
PostgreSQL, SQLAlchemy, Alembic, наблюдения объявлений.
ML
Non-leaky features, joblib artifacts, сравнение кандидатов.
API
FastAPI endpoints для данных, модели и мониторинга.
UI
Streamlit dashboard на русском языке для защиты.
Raw listings → Normalization → PostgreSQL → FastAPI → Streamlit ↘ ML feature snapshot → trained model artifacts → /predict

Интерпретация признаков

Что означают поля окружения в форме оценки

Поля «школы 1 км», «парки 1 км», «транспорт 500 м» — это счётчики объектов инфраструктуры в заданном радиусе. Изменение на единицу означает один дополнительный найденный объект, а не один процент и не один балл.

UI/UX

Интерфейс для демонстрации и проверки

Streamlit dashboard показывает полный контур проекта: от рыночных данных до объяснимого результата модели. Интерфейс работает с реальным API, а не с фиктивным mock.

Оценка квартиры

Пользователь меняет площадь, комнаты, этаж, год, координаты и окружение; API пересчитывает стоимость по модели.

Карта

Точки объявлений, масштабирование, перемещение карты, popup с ценой и ссылкой на источник.

Мониторинг

Статусы API, БД, Redis, модели, ingestion, источников и покрытия OSM-признаками.

Запуск и развёртывание

Одинаковая логика локально и на VPS

Проект упакован в Docker Compose. Для VPS используются production compose, Caddy reverse proxy, PostgreSQL volume и отдельный volume с артефактами модели.

Локально

docker compose -p realtyscope up -d --build curl -fsS http://localhost:8000/health curl -fsS http://localhost:8501/_stcore/health

VPS

cd /opt/realtyscope git pull bash scripts/deployment/restore_vps_runtime_bundle.sh docker compose -f docker-compose.prod.yml up -d --build
Готовность к защите

Что важно проговорить комиссии

Сильные стороны

  • end-to-end pipeline: данные → БД → API → UI → ML;
  • реальные данные и проверяемые runtime-метрики;
  • модель без прямой price leakage;
  • Docker/VPS-ready упаковка;
  • русскоязычный UI для демонстрации.

Честные ограничения

  • это учебный оценочный snapshot, не промышленная экспертиза стоимости;
  • random validation для недвижимости может быть оптимистичной;
  • lifecycle forecast основан на observation gaps, а не на подтверждённых продажах;
  • модельные артефакты и БД восстанавливаются через runtime bundle, а не хранятся в Git.
Короткая защита: RealtyScope показывает полный цикл инженерии данных и ML-сервиса. Главная ценность проекта — не только R² модели, а воспроизводимый backend/UI контур, честная валидация и готовность к запуску через Docker/VPS.
